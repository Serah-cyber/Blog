Investigating Suspicious ICMP Traffic in a SOC Lab Using Wireshark
#

Alert Summary
#

The SIEM generated a medium-severity alert for unusual ICMP traffic between two internal hosts.
The alert indicated:

  • Large ICMP packet sizes
  • Repeated echo requests
  • High frequency within a short time window

Possible concern: Data exfiltration over ICMP (ICMP tunneling).

Environment
#

  • Tool : Wireshark
  • Data Source: Provided PCAP file from SOC lab
  • Alert Type: Network anomaly detection
  • Protocol: ICMP

Step 1 — Identify ICMP Traffic
#

Open the PCAP file in Wireshark.

Apply the following display filter:

icmp

This isolates all ICMP traffic.

Observe:

  • Source IP
  • Destination IP
  • Type (Echo request / reply)
  • Packet length

Step 2 — Check ICMP Packet Types
#

To focus only on echo requests:

Run:

icmp.type = = 8

Echo reply:

icmp.type = = 0

In this case, I observed a high number of echo requests from:

192.168.1.25 → 192.168.1.10

The volume appeared abnormal compared to normal ping behavior.

Step 3 — Investigate Packet Size
#

Large ICMP packets can indicate tunneling or embedded data.

Apply:

icmp && frame.len > 200

Normal ping packets are typically small (around 64–98 bytes).

However, I observed multiple packets exceeding 1000 bytes.

This is a strong anomaly indicator.

Step 4 — Analyze Payload Content
#

Click on one of the large ICMP packets.

Expand:

  • Internet Control Message Protocol
  • Data section

In normal pings, payloads are predictable or random patterns.

In this case:

  • Payload length was unusually large
  • Repetitive structured patterns were visible
  • Data did not resemble normal ping behavior

This suggests possible encoded or tunneled data.

Step 5 — Check Traffic Pattern
#

To analyze frequency:

Statistics → Conversations → ICMP

I observed:

  • High packet count
  • Continuous flow
  • Not typical diagnostic behavior

Legitimate ping activity is usually:

  • Short bursts
  • Low frequency
  • Limited duration

This traffic was persistent.

Step 6 — Rule Out False Positives
#

Possible legitimate causes:

  • Network monitoring tools
  • Large file transfer testing
  • Misconfigured internal tools

No documentation indicated approved ICMP-based tools.

Given:

  • Large packet size
  • High frequency
  • Structured payload

The activity is suspicious.

Verdict
#

This activity is highly indicative of:

  • ICMP tunneling
  • Potential data exfiltration attempt

The alert is likely a True Positive.

Escalation to incident response is recommended.

MITRE ATT&CK Mapping
#

  • T1048 — Exfiltration Over Alternative Protocol
  • T1095 — Non-Application Layer Protocol

Lessons Learned
#

  • Packet size matters in anomaly detection.
  • Not all ICMP traffic is benign.
  • Frequency + size + payload structure together indicate malicious intent.
  • Wireshark filtering is essential for rapid triage.

Blue Team Takeaway
#

ICMP is often overlooked because it is considered harmless.

However, attackers may abuse it for:

  • Data exfiltration
  • Command-and-control
  • Network reconnaissance

Blue teams should implement detection rules for:

  • Abnormally large ICMP packets
  • High-frequency echo requests
  • Persistent ICMP sessions
